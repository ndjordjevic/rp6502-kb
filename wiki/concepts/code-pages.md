# Code Pages

**Summary**: Code pages are numbered 8-bit character encodings that control both the glyph set displayed on screen and the short-name encoding used by the FAT filesystem driver — allowing the 6502 to work in a specific language's character set.

---

## Background

The IBM PC 5150 (1981) introduced a 256-glyph set — the lower 128 glyphs were ASCII; the upper 128 combined graphics characters and accented letters selected from Wang word processor character sets. This set became known as **code page 437**. Subsequent generations of video cards allowed loading fonts from RAM, and regional variants proliferated into a standardized family that eventually converged on Unicode.

The Picocomputer uses code pages because they fit the memory and simplicity constraints of 8-bit computing while still supporting modern workflows (modern USB drives, host file transfer).

## Supported code pages

| Code page | Common name | Coverage |
|---|---|---|
| **CP437** | OEM United States (IBM PC original) | English, German, Swedish; box-drawing characters |
| **CP850** | Latin-1 / Multilingual (Picocomputer **default**) | Western European languages; French, Spanish, Portuguese, etc. |
| **CP855** | Cyrillic | Eastern European languages using Cyrillic script |

## What selecting a code page does

Selecting a code page has **two simultaneous effects**:

1. **Swaps the glyph set** displayed by the console/VGA terminal — the upper 128 character positions now render according to the selected code page.
2. **Tells the FAT driver** which short-name encoding to use when reading 8.3 filenames from removable media.

## FAT short-name fallback behavior

Modern FAT filesystems store both a long Unicode name (UCS-2) and a legacy 8.3 short name for each file. The 8.3 name uses a single-byte encoding determined by the active code page.

**What happens with a mismatch:**

If a file's long name contains characters that are not representable in the active code page, the 8.3 short name is the only way to access it from the 6502 side. The short name is generated by the system that created the file and will have a `~1` suffix (e.g., `MYFILE~1.TXT`).

**Example from Ep12**:
- With CP850 active: a French filename (`résumé.txt`) displays correctly; a Cyrillic filename falls back to its 8.3 form (e.g., `RESUMÉ~1.TXT`).
- Switch to CP855: the Cyrillic filename is revealed; the French filename now falls back to its 8.3 form.

The `~1` is not an error — the file is still accessible, but the name is opaque until you switch to the matching code page.

## Setting the code page

From the monitor:
```
SET CODEPAGE 437
SET CODEPAGE 850
SET CODEPAGE 855
```

From a program: use the `RIA_ATTR_CODE_PAGE` attribute via `ria_set_attr()` (see [[rp6502-os]]).

## Interaction with custom fonts

The code page setting changes the **system font** (the console's built-in character ROM equivalent). Application programs can override this with custom fonts — the code page then only affects the file system interaction, not the displayed glyphs.

## Keyboard support note

As of Ep12, only US keyboard layout was supported. Non-US keyboard input (diacritics via hardware keyboard) requires additional driver work.
