# RP2040 SPI

**Summary**: The RP2040 has two SPI peripherals (SPI0, SPI1) supporting master/slave mode, 4–16-bit words, all four CPOL/CPHA modes, DMA, and interrupts; clock derived from `clk_peri`. The RIA firmware does not use SPI — storage is via USB MSC + FatFS.

---

## SPI Basics

SPI (Serial Peripheral Interface) is a full-duplex synchronous serial protocol. One bit is sent and one bit is received simultaneously on every clock pulse. The master generates the clock and controls slave selection.

**Topology**: master/slave multi-drop — all slaves share SCLK/MOSI/MISO; each slave has its own SS̄ (active-low select). Multiple slaves can coexist on the same bus.

**3-wire SPI** (half-duplex, MOSI and MISO combined) is **not supported** by the RP2040 SPI hardware but can be implemented with PIO.

---

## Signals

| Signal | Direction (master) | Description |
|---|---|---|
| **SCLK** | Output | Serial clock generated by the master |
| **MOSI** | Output | Master Out Slave In — data from master to slave |
| **MISO** | Input | Master In Slave Out — data from slave to master |
| **SS̄** | Output | Slave Select — active low; each slave has its own line |

Alternative names in datasheets/code: SCLK=SSPCLK/SCK, MISO=SSPRXD/RX, MOSI=SSPTXD/TX, SS=SSPFSS/CSN.

---

## SPI Modes

Four modes based on clock polarity (**CPOL**) and clock phase (**CPHA**):

| Mode | CPOL | CPHA | SCLK idle | Data captured on |
|---|---|---|---|---|
| 0 | 0 | 0 | LOW | Leading (rising) edge |
| 1 | 0 | 1 | LOW | Trailing (falling) edge |
| 2 | 1 | 0 | HIGH | Leading (falling) edge |
| 3 | 1 | 1 | HIGH | Trailing (rising) edge |

CPHA=0: data changes on trailing edge of previous cycle (or before first), captured on leading edge of current cycle. CPHA=1: reversed.

Most devices use Mode 0 or Mode 3. Example: the ADXL345 accelerometer uses Mode 3 (CPOL=1, CPHA=1).

---

## RP2040 SPI Peripheral Features

Two SPI peripherals (SPI0, SPI1):
- Master or slave mode
- Data word size: **4 to 16 bits**
- **8-entry TX FIFO + 8-entry RX FIFO**
- All four SPI modes (CPOL + CPHA)
- Flexible clock generation
- Interrupt and DMA support
- Also supports TI synchronous serial and National Semiconductor Microwire (not commonly needed)

---

## Clock Generation (Master Mode)

The SPI clock is derived from `clk_peri` through a **two-stage integer divisor**:
1. Stage 1: divides `clk_peri` by an **even number** from 2 to 254 (steps of 2)
2. Stage 2: divides the result by a value from 1 to 256

`spi_init()` / `spi_set_baudrate()` compute the closest achievable frequency automatically and return the actual value.

---

## Slave Select (SS) in Master Mode

**The SPI peripheral does NOT automatically control the SS̄ pin in master mode.** You must:
1. Configure a regular GPIO as output (initially HIGH / deasserted)
2. Drive it LOW (`gpio_put(ss_pin, 0)`) before a transfer
3. Drive it HIGH (`gpio_put(ss_pin, 1)`) after the transfer completes

Add brief NOP delays (`asm volatile("nop")`) around SS transitions if the slave requires setup/hold time. In slave mode the peripheral uses its SS pin automatically.

---

## GPIO Pin Options

| Function | SPI0 GPIOs | SPI1 GPIOs |
|---|---|---|
| SCLK | 2, 6, 18, 22 | 10, 14, **26** |
| MISO (RX) | 0, 4, 16, 20 | 8, 12, 24, 28 |
| MOSI (TX) | 3, 7, 19, 23 | 11, 15, 27 |
| SS (slave mode) | 1, 5, 17, 21 | 9, 13, 25, 29 |

GPIO function select: `GPIO_FUNC_SPI`. Remember: SS is only relevant if operating in **slave mode**; in master mode use any GPIO.

## SDK (`hardware_spi`)

| Function | Description |
|---|---|
| `spi_init(spi, baudrate)` | Init in master mode; returns actual baud rate |
| `spi_set_baudrate(spi, baud)` | Change baud rate; returns actual value |
| `spi_set_format(spi, data_bits, cpol, cpha, order)` | Format: data_bits 4–16, cpol/cpha constants, order must be `SPI_MSB_FIRST` |
| `spi_set_slave(spi, slave)` | `false` = master (default), `true` = slave |
| `spi_is_writable(spi)` | Space available in TX FIFO |
| `spi_is_readable(spi)` | Data available in RX FIFO |
| `spi_is_busy(spi)` | Transmitting or TX FIFO non-empty |
| `spi_write_read_blocking(spi, src, dst, len)` | Full-duplex transfer, 8-bit buffers |
| `spi_write16_read16_blocking(spi, src, dst, len)` | Full-duplex, 16-bit buffers |
| `spi_write_blocking(spi, src, len)` | Send only, discard RX (8-bit) |
| `spi_write16_blocking(spi, src, len)` | Send only, discard RX (16-bit) |
| `spi_read_blocking(spi, repeated_tx, dst, len)` | Receive only; sends `repeated_tx` repeatedly (8-bit) |
| `spi_read16_blocking(spi, repeated_tx, dst, len)` | Receive only (16-bit) |
| `spi_get_dreq(spi, is_tx)` | DMA DREQ: `is_tx=true` for TX, `false` for RX |

All blocking functions return the number of items transferred.

### Typical initialization sequence

```c
spi_init(spi0, 1000000);                               // 1 MHz
spi_set_format(spi0, 8, SPI_CPOL_1, SPI_CPHA_1, SPI_MSB_FIRST);  // Mode 3
gpio_set_function(SCK_PIN,  GPIO_FUNC_SPI);
gpio_set_function(MISO_PIN, GPIO_FUNC_SPI);
gpio_set_function(MOSI_PIN, GPIO_FUNC_SPI);
gpio_init(SS_PIN); gpio_set_dir(SS_PIN, GPIO_OUT); gpio_put(SS_PIN, 1);
```

## RIA Firmware Note

The ingest plan listed "SPI: SD card storage" but the actual RIA firmware uses **USB Mass Storage Class (USB MSC) + FatFS** for all file storage. USB thumb drives (or devices exposing MSC) are mounted automatically. The SPI peripheral is not used by the current RIA firmware — `clk_peri` drives it but the SPI hardware sits idle.

See [[usb-controller]] for the TinyUSB MSC host implementation and [[rp2040-clocks]] for `clk_peri` details.
