# SDK Architecture

**Summary**: The Pico SDK build model — CMake INTERFACE libraries, library naming tiers, hardware claiming, the builder pattern, and the RP2040/RP2350 platform split — foundational context for reading RIA and VGA firmware source.

---

## Build system

The SDK uses **CMake** exclusively. Every application is a C/C++ source file plus a `CMakeLists.txt`:

```cmake
add_executable(my_app my_app.c)
target_link_libraries(my_app pico_stdlib)
pico_add_extra_outputs(my_app)   # produces .uf2, .bin, .hex, .map, .dis
```

`pico_add_extra_outputs` calls `picotool` to produce the UF2 file for USB mass-storage flashing. ELF files can also be loaded directly via SWD with `gdb` + `openocd` or `picotool`.

`main()` return value is ignored by the SDK runtime — the processor hangs on exit.

## INTERFACE library model

Every SDK library is a CMake **INTERFACE library**: a bundle of source files, include paths, compiler definitions, and link options. These form a dependency tree.

Key consequence: **all code for one executable is recompiled from source**. This enables per-application compile-time customisation:
- Enable asserts (`PARAM_ASSERTIONS_ENABLE_ALL=1`)
- Set static buffer sizes via `#define`
- Linker GC removes every uncalled function

```cmake
target_link_libraries(my_app pico_stdlib hardware_dma)
```

`hardware_dma` must be listed explicitly — it is not included in `pico_stdlib`. Without the dependency, `#include "hardware/dma.h"` will not compile.

## Library naming tiers

| Prefix | Examples | Role |
|---|---|---|
| `pico_` | `pico_stdlib`, `pico_multicore`, `pico_sync`, `pico_time` | Higher-level; cross-cutting concerns (timers, sync, multi-core) |
| `hardware_` | `hardware_pio`, `hardware_dma`, `hardware_gpio`, `hardware_clocks` | Thin peripheral wrappers; near-zero RAM; only depend on `hardware_structs` + `hardware_regs` |
| `hardware_structs` | — | C structs over memory-mapped register blocks |
| `hardware_regs` | — | Autogenerated `#define` + comment headers; includable from `.S` assembly |

Naming convention: library `foo_bar` → header `#include "foo/bar.h"`. Example: `hardware_uart` → `#include "hardware/uart.h"`.

`pico_stdlib` aggregates `pico_runtime`, `hardware_gpio`, `hardware_uart`, and others as implicit dependencies. `hardware_dma`, `hardware_pio`, etc. must be added explicitly.

## Hardware structs and atomic aliases

`hardware_structs` maps peripheral register blocks to typed C structs:

```c
// Raw register access (verbose, error-prone):
*(volatile uint32_t *)(PIO0_BASE + PIO_SM1_SHIFTCTRL_OFFSET) |= PIO_SM1_SHIFTCTRL_AUTOPULL_BITS;

// Struct access (readable, compiler-friendly):
pio0->sm[1].shiftctrl |= PIO_SM1_SHIFTCTRL_AUTOPULL_BITS;
```

RP-series peripherals expose three address aliases per register — **set**, **clear**, **XOR** — that perform atomic bit manipulation. These are a **hardware feature, not CPU-emulated**: DMA can also perform atomic register operations via aliases.

```c
hw_set_alias(pio0)->sm[1].shiftctrl = PIO_SM1_SHIFTCTRL_AUTOPULL_BITS;  // atomic OR
hw_set_bits(&pio0->sm[1].shiftctrl, PIO_SM1_SHIFTCTRL_AUTOPULL_BITS);   // equivalent helper
```

Used throughout the SDK to avoid data races when two cores or an IRQ and foreground code access the same register concurrently. Directly relevant to the RIA dual-core PIO access pattern.

## Hardware claiming

`hardware_xxx` libraries include a runtime claiming mechanism: `pio_claim_sm_mask()`, `dma_channel_claim()`, etc. Claiming registers a resource as in-use atomically. Attempting to claim an already-claimed resource triggers a `panic()` in debug builds. Prevents silent double-use of a DMA channel or PIO state machine by two independent libraries.

## Builder pattern for peripheral config

Complex peripheral configurations use a **builder pattern** to avoid large argument lists and to exploit constant folding:

```c
int dma_ch = 3;
dma_channel_config c = dma_get_default_channel_config(dma_ch);
channel_config_set_read_increment(&c, true);
channel_config_set_write_increment(&c, true);
channel_config_set_dreq(&c, DREQ_SPI0_RX);
channel_config_set_transfer_data_size(&c, DMA_SIZE_8);
dma_set_config(dma_ch, &c, false);
```

Because `channel_config_set_*` functions are `static inline` and `dma_ch` is a compile-time constant, the compiler constant-folds everything into **a single register write**:

```c
*(volatile uint32_t *)(DMA_BASE + DMA_CH3_AL1_CTRL_OFFSET) = 0x00089831;
```

This pattern is used for DMA channel config, PIO state machine config, clock configuration, and GPIO function selection. The struct acts as an intermediate accumulator that prevents the compiler from emitting repeated volatile register accesses.

## Function naming conventions

- **Prefix** = library/area: `dma_channel_`, `pio_sm_`, `gpio_`, `clock_`
- **Verb**: `set_`, `get_`, `is_` — always present
- **Blocking suffixes**:

| Suffix | Param type | Meaning |
|---|---|---|
| (none) | — | Non-blocking |
| `_blocking` | — | May block indefinitely |
| `_blocking_until` | `absolute_time_t until` | Block until condition or timeout |
| `_timeout_ms` | `uint32_t timeout_ms` | Block up to N milliseconds |
| `_timeout_us` | `uint64_t timeout_us` | Block up to N microseconds |

## Error handling

- **Legitimate runtime failures** (timeout, resource exhausted): return `bool` or `PICO_ERROR_*` (negative) / `PICO_OK` (0)
- **Programming bugs** (wrong channel number, double-claim): `assert()` — disabled in release builds by default; `PARAM_ASSERTIONS_ENABLE_ALL=1` to enable globally
- **Unrecoverable state**: `panic()` — writes message to stdio then halts with a breakpoint instruction

## Directory and platform split

| Directory | Contents |
|---|---|
| `src/rp2040/` | RP2040-specific `hardware_regs` + `hardware_structs` |
| `src/rp2350/` | RP2350-specific ditto |
| `src/rp2_common/` | Shared `hardware_xxx` and `pico_xxx` implementations for all RP-series |
| `src/common/` | Platform-neutral utility code |
| `src/host/` | Stub implementations for host-mode testing (Linux/macOS/Windows) |

Controlled by `PICO_PLATFORM`: `rp2040`, `rp2350-arm-s`, `rp2350-riscv`. Setting `PICO_BOARD` can auto-set `PICO_PLATFORM`.

**Both RIA and VGA firmwares target RP2350** (Pi Pico 2), so the `rp2350-arm-s` platform applies to both. RP2350-exclusive libraries (DCP for double-precision float, `hardware_hazard3`, `hardware_powman`) are available but not used in the current RP6502 firmware.

## Multi-core model

- Core 1 started from core 0 via `multicore_launch_core1(fn)` (`pico_multicore`)
- Inter-core mailbox FIFOs for lightweight message passing
- C standard library functions are **not** thread-safe by default; protect with `mutex_` from `pico_sync`
- `malloc`/`calloc`/`free` are wrapped to be thread-safe when `pico_multicore` is linked
- SDK `printf` is always safe to call from both cores

See [[dual-core-sio]] for SIO hardware details and [[rp6502-ria]] for the RIA two-core split.

## Runtime

`pico_stdlib` pulls in three runtime libraries:
- `pico_crt0` — entry point, default linker scripts, memory layout
- `pico_standard_link` — linker options and script selection
- `pico_runtime_init` — initializers run before `main()`

Default stdio: UART at 115200 baud on **GPIO 0 (TX) / GPIO 1 (RX)**; USB CDC ACM; Segger RTT; semihosting. The RIA firmware overrides this to **GPIO 4 (TX) / GPIO 5 (RX)** via `PICO_DEFAULT_UART_TX_PIN`/`PICO_DEFAULT_UART_RX_PIN` in its board config. See [[gpio-pinout]].

## Floating point

- **RP2040**: single- and double-precision via bootrom ROM routines. Single-cycle access via dedicated arbiter; zero flash and binary-size cost.
- **RP2350**: single-precision via Arm VFP hardware instructions; double-precision via RP2350 DCP (Double Co-Processor). Substantially faster than RP2040.

RP2040 FP note: if the firmware targets RP2040 boards with bootrom V2 only, set `PICO_FLOAT_SUPPORT_ROM_V1=0` and `PICO_DOUBLE_SUPPORT_ROM_V1=0` to exclude V1-compatibility stubs.

## Board customisation

Per-board defaults come from a board config header selected by `PICO_BOARD`. Per-binary overrides:

```cmake
target_compile_definitions(my_app PRIVATE
    PICO_DEFAULT_UART_TX_PIN=4
    PICO_DEFAULT_UART_RX_PIN=5
)
```

All SDK preprocessor defines have a `PICO_` prefix.

## C++ support

SDK headers declare functions with C linkage; safe to include from C++. C++ exceptions disabled by default (`PICO_CXX_ENABLE_EXCEPTIONS=1` to enable).
