# cc65

**Summary**: cc65 is the primary C cross-development package for the RP6502 — a C compiler, macro assembler, and linker targeting the 6502/65C02, in use since 1998 with a mature standard library and a VSCode template project for the Picocomputer.

---

## What it is

cc65 is a complete cross-development package for 6502-family processors:
- **C compiler** — targeting 6502/65C02 (no floating point; ints are 16-bit; `long` is 32-bit)
- **Macro assembler** (`ca65`)
- **Linker** (`ld65`)
- **Standard libraries** — `stdio.h`, `unistd.h`, `fcntl.h`, `stdlib.h`, `time.h`, etc.

The RP6502 cc65 port provides wrappers that map standard POSIX functions to [[rp6502-os]] calls transparently.

## Relationship to RP6502 ABI

The [[rp6502-abi]] is explicitly modeled on the **cc65 fastcall convention**:
- Last argument in A/X registers; remaining args pushed left-to-right on XSTACK.
- `RIA_ERRNO` maps to the cc65 `errno` variable.
- The `randomize()` function (cc65 extension) uses hardware entropy for seeding the RNG.

## VSCode development environment

From [[yt-ep09-c-programming-setup]]:
- Template repository on GitHub: create from template → clone → init submodule.
- **Ctrl+Shift+B** in VSCode: builds project, packages as `.rp6502` ROM, uploads to Picocomputer over USB, issues reset to run.
- `rp6502.py`: Python script underlying the VSCode task; also usable directly for upload/run/ROM operations.
- `config.ld`: controls 6502 memory layout; rarely needs changing.
- cc65 IntelliSense hacks included in template — do not override with IntelliSense setup.

## Known fork requirement

As of v0.14 (Oct 2025): use the official fork at `github.com/picocomputer/cc65` until upstream PR #2844 (errno rework) merges into the main cc65 project. The standard upstream cc65 does not yet include the matching errno system.

## Comparison with LLVM-MOS

| Aspect | cc65 | [[llvm-mos]] |
|---|---|---|
| Maturity | Since 1998; stable | Newer; less stable |
| Standard library | Complete | Sparse (growing) |
| C++ support | No | Yes |
| Float / 64-bit | No | Yes |
| Code size | ~equivalent (large apps) | ~equivalent (large apps) |
| Performance | Good with idiomatic cc65 code | Better optimization out-of-the-box |
| Best for | Most projects needing stdlib | C++, floats, or 64-bit math |

From [[yt-ep18-llvm-mos]] (Ep18): LLVM-MOS gave a significant Mandelbrot speed improvement without special effort, while cc65 can be made comparable but requires crafting code specifically for its optimizer.

## C language limitations

- **No `float`** (no floating-point support)
- **No 64-bit integers** (`long long` not supported)
- **16-bit `int`**: when AI assistants write cc65 code, they must be told this explicitly or they attempt 64-bit fixed-point math (from [[yt-ep21-ai-programming]])
