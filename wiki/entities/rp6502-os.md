# RP6502-OS

**Summary**: The 32-bit protected operating system that runs **inside** the [[rp6502-ria]] and exposes a POSIX-like API to the 6502 program. Uses **zero** 6502 RAM and never interferes with developing your own native 6502 OS on top.

---

## What "OS" means here

This is **not** a 6502 OS. It is a host service running on the RP2350 inside the RIA, callable by the 6502 through the [[rp6502-abi]]. From the 6502's perspective it looks like a tray of well-behaved POSIX functions: open files, talk to USB devices, get the time, install virtual hardware on the [[pix-bus]].

The 6502 has no ROM. Zero page is unreserved. You can ignore the OS entirely and write a from-scratch 6502 monitor; the OS will not get in your way.

## What it provides

- **Filesystem** — FAT32 (FatFs under the hood; ExFAT "ready when patents expire"). Up to 8 files + 8 dirs open at once.
- **stdio / unistd / fcntl / time / stdlib** — for both [[cc65]] and [[llvm-mos]] runtimes.
- **`xreg()` / `xregn()`** — install virtual hardware on the PIX bus. See [[xreg]].
- **Console / VCP / NFC / "ROM:"** — exposed as openable filenames (`CON:`, `TTY:`, `VCP0:115200,8N1`, `NFC:`, `ROM:asset_name`).
- **RIA Attributes** — `ria_attr_get` / `ria_attr_set` for runtime knobs:
  | ID | Name | Effect |
  | --- | --- | --- |
  | 0x00 | `ERRNO_OPT` | Pick errno table (cc65 / llvm-mos) |
  | 0x01 | `PHI2_KHZ` | CPU clock 100–8000 kHz; reverts on ROM exit |
  | 0x02 | `CODE_PAGE` | Active OEM code page; reverts on ROM exit |
  | 0x03 | `RLN_LENGTH` | stdin line-editor max length |
  | 0x04 | `LRAND` | 31-bit hardware random |
  | 0x05 | `BEL` | Console BEL on/off |
  | 0x06 | `LAUNCHER` | Register self as [[launcher]] |
  | 0x07 | `EXIT_CODE` | (experimental) last ROM's exit code |
- **Process** — `_argv`, `_exec` / `ria_execl` / `ria_execv`, `exit`, [[launcher]] indirection.
- **Time** — `clock`, `clock_gettime`, `clock_settime`, `tzset`, `localtime`. Monotonic clock is 100 Hz, wraps ~497 days.

## Errno mapping

OS calls set `RIA_ERRNO` (`$FFED-$FFEE`). The OS maps both **FatFs** and **LittleFS** error codes onto its 19-entry internal errno enum, then translates to the platform encoding. Both compilers' errno tables are supported via `RIA_ATTR_ERRNO_OPT` (set automatically by their C runtime startup). Full error list in [[ria-registers]].

## Complete API surface (by op-code group)

See [[api-opcodes]] for the full numbered table. Groups:
- **0x01–0x0B** — system attributes (xreg, phi2, code page, random, errno-opt, argv, exec, attr get/set)
- **0x0D–0x12** — time (tzset, tzquery, clock, clock_getres, clock_gettime, clock_settime)
- **0x14–0x1E** — file I/O (open, close, read×2, write×2, lseek×2, syncfs)
- **0x1B–0x2E** — filesystem (unlink, rename, stat, opendir, readdir, closedir, telldir, seekdir, rewinddir, chmod, utime, mkdir, chdir, chdrive, getcwd, setlabel, getlabel, getfree)

## Input modes (from [[yt-ep14-usb-mouse]])

> **Source**: [[yt-ep14-usb-mouse]] (Ep14). The three-mode input system evolved over the first year of development.

### Mode 1: UART stdio (console)

The original input mode. `printf()` and `fgets()` over standard I/O — same as any POSIX terminal.

`fgets()` with a line editor was **added in the release associated with Ep14**, completing the stdio input system. It provides line editing (backspace, etc.) from the keyboard before returning to the program.

### Mode 2: HID keyboard bit-array

A 256-bit array in XRAM where each bit corresponds to a HID keycode:
- Bit = 1: key is currently pressed (held)
- Bit = 0: key is released
- **Bit 0**: special — set when *no* keys are pressed. Useful for "wait for any key" without needing to know which key.

Modifiers (Shift, Ctrl, etc.) are also detected. This mode enables smooth game controls with held-key detection.

**Setup**: write the XRAM start address of the bit array to `$0:0:00` xreg. Then poll the bits in XRAM using the RIA address registers.

### Mode 3: Mouse integer counters

Mouse movement is accumulated as signed integers in XRAM:
- X counter increases/decreases with horizontal movement.
- Y counter increases/decreases with vertical movement.
- Button states also available.

**Usage pattern**: read current counter → subtract previous counter value → delta is mouse movement since last check. Reset previous to current.

**Setup**: write the XRAM start address to `$0:0:01` xreg.

All USB mice (including wireless optical mice) work through the standard USB HID stack — no special drivers.

## Design philosophy (from [[yt-ep07-operating-system]])

> **"I never set a goal of making an operating system."** — The OS emerged from wanting to run POSIX-like software. From [[yt-ep07-operating-system]] (Ep7, 2023): *"Turns out if you want to run POSIX-like software you'll eventually find yourself writing the kernel for a POSIX-like operating system."*

Key framing from Ep7:

- **"All I ask for is 32 bytes"** — the RIA's only claim on the 6502 address space (`$FFE0–$FFFF`). Every other byte is free for user programs.
- **Protection through isolation** — kernel calls go through the RIA hardware interface; a crashing 6502 program cannot bring down the kernel. No memory manager required.
- **OS is optional** — over 1 MB with networking + code pages; "completely vanishes" when not called.
- **Third-party composition** — FatFs, littlefs, TinyUSB form the bulk of the OS. The author wrote the device drivers and integration layer.
